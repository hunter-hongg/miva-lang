# Night Shift Task: Add TOML and YAML stdlib libraries to Miva

You are working in the Miva language compiler repo (`~/projects/miva-lang`), a multi-crate
Rust workspace that compiles Miva source to native binaries (C++/LLVM/MVM backends).

## Goal

Add two new standard-library modules — **TOML** and **YAML** — by closely mirroring
the existing **JSON** and **XML** implementations. Match their structure, naming
conventions, memory-ownership model, and code style exactly.

## Study these references FIRST (read them fully before writing anything)

1. `stdlib/std-0.1.1/src/json.miva`  — the Miva stdlib wrapper (opaque handle over runtime builtins)
2. `stdlib/std-0.1.1/src/xml.miva`   — same pattern for XML
3. Runtime builtins in Rust:
   - `miva-vm/src/xml.rs`             — how XML runtime tree/handles are implemented
   - Search for `json_parse`, `json_kind`, `json_*`, `xml_*` across:
     `miva/src/symbol_table.rs`, `miva/src/typecheck.rs`,
     `miva/src/codegen/cxx.rs`, `miva/src/codegen/llvm.rs`, `miva/src/codegen/mvm.rs`,
     `miva-vm/src/vm.rs`
4. Examples: `examples/json/` and `examples/xml/` (miva.toml + src/main.miva)

## Deliverables

1. `stdlib/std-0.1.1/src/toml.miva` — Miva wrapper module `std.toml`, mirroring json.miva's
   API surface where it makes sense (parse, kind predicates, accessors, len, iteration,
   serialize/stringify, free semantics with the same borrowed-view ownership rules).
2. `stdlib/std-0.1.1/src/yaml.miva` — Miva wrapper module `std.yaml`, same pattern.
3. Runtime builtins in Rust so the new modules actually work:
   - Register `toml_*` and `yaml_*` builtin functions everywhere the `json_*`/`xml_*`
     ones are registered (symbol_table, typecheck, all three codegen backends, vm).
   - Implement the parsing/serialization runtime, mirroring how `miva-vm/src/xml.rs`
     (and the json runtime) work. Add `miva-vm/src/toml.rs` and `miva-vm/src/yaml.rs`
     if that matches the existing file organization; wire them into the module tree.
   - If an external crate is the pragmatic choice (e.g. `toml`, `serde_yaml`/`serde_yml`),
     add it to the relevant `Cargo.toml` and use it, but keep the handle/tree/ownership
     model identical to JSON/XML.
4. `examples/toml/` and `examples/yaml/` — copy the shape of `examples/json/`
   (a `miva.toml` project file + `src/main.miva` that parses a sample document,
   reads a few values, and prints them). Do NOT commit build artifacts.
5. Tests: add `.miva` test files under `tests/` following the existing json/xml test style.

## Constraints

- Match existing code style, naming, comments, and safety annotations (safe/unsafe/trusted).
- Preserve the memory-ownership rules from json.miva (only the root handle owns memory;
  array/object accessors return borrowed views that must not be freed).
- Do NOT break existing JSON/XML functionality.

1. `cargo build --release` succeeds for the whole workspace.
2. Build and run BOTH new examples end-to-end, e.g.:
   `cd examples/toml && cargo run --manifest-path ../../miva/Cargo.toml -- --build --release && ... --run --release`
   (use whatever the correct invocation is — infer it from how examples/json is built;
    check `build.sh` and `examples/json/miva.toml`).
3. Confirm the TOML and YAML examples produce correct parsed output.
4. Run the existing test suite (`tests/run_tests.sh`) and confirm no regressions.
